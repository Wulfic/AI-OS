WHAT EXISTS
-----------

- config/mcp_servers.json (placeholder server entries)

- src/aios/core/mcp/tool_executor.py (parses tool calls and returns mock responses)
- Chat UI instantiates the tool executor so tools appear in the prompt context

AVAILABLE MOCK TOOLS (8)
------------------------

Filesystem:
  read_file, write_file, list_directory, search_files

Memory:
  create_entities, search_nodes

DuckDuckGo:
  web_search, fetch_webpage

All executions produce placeholder messages such as:
  "[Mock] Tool read_file would be executed with: {...}"

SUPPORTED TOOL CALL FORMATS
---------------------------

1. JSON: {"tool": "read_file", "parameters": {"path": "README.md"}}
2. XML-style: <tool_call>{"name": "read_file", "arguments": {"path": "README.md"}}</tool_call>
3. Function-style: read_file(path="README.md")

LIMITATIONS
-----------

- No MCP servers are spawned or contacted.
- Node.js/npm are not required yet.
- No scripts or pytest suites ship for MCP verification.
- Tool results are always mocked; real file/web access does not occur.

HOW TO EXPERIMENT
-----------------

```python

executor = ToolExecutor()
print(executor.get_system_prompt_addition())
```

Use the chat GUI to trigger tool formatting, but expect mock outputs.
